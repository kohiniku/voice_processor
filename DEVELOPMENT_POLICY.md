# Development Policy Template (Reusable)

このドキュメントは、別プロジェクトでも同じ開発品質を再現するための「汎用テンプレート」です。  
固有名詞（製品名・モデル名・特定パス）は入れず、どの案件にも適用できる粒度で定義します。

## 1. 各種技術スタック（テンプレート）

以下のカテゴリを必ず定義し、採用理由を1行で記録する。

- 言語・ランタイム
  - 例: `Language`, `Runtime Version`
- アプリケーションフレームワーク
  - 例: Web/API Framework, CLI Framework
- インフラ・実行基盤
  - 例: Container, Orchestrator, GPU/CPU 方針
- 設定・シークレット管理
  - 例: `.env`, Secret Manager, 型付き設定
- データ/外部連携
  - 例: DB, Queue, Object Storage, 外部API
- 品質管理
  - 例: Test Framework, Linter, Formatter, Type Checker
- CI/CD
  - 例: Build, Test, Deploy の自動化手段
- 監視・運用
  - 例: Logging, Metrics, Tracing, Alerting

記載フォーマット例:

```md
- 言語・ランタイム: <name/version>（理由: <one-line>）
- Web/API: <framework>（理由: <one-line>）
- テスト: <tool>（理由: <one-line>）
```

## 2. フォルダ・ファイル構成（テンプレート）

責務境界が明確になる構成を採用する。最低限、次のレイヤを分離する。

```text
project-root/
├── src/                  # 本体コード
│   ├── api/              # 入出力境界（HTTP/CLI/Message）
│   ├── domain/           # ドメインモデル・業務ルール
│   ├── services/         # ユースケース実装
│   ├── infra/            # DB/API/外部サービス接続
│   └── config/           # 設定ロード・DI配線
├── tests/
│   ├── unit/
│   ├── integration/
│   └── e2e/
├── scripts/              # 運用補助スクリプト
├── docs/                 # 設計・運用ドキュメント
├── Dockerfile
├── compose.yml
├── .env.example
├── pyproject.toml or package.json
└── README.md
```

構成ルール:

- `api` は薄く保ち、ビジネスロジックは `services` / `domain` に置く
- 環境依存処理は `infra` に寄せる
- 設定値は `config` 経由で注入し、直接参照を減らす
- テストディレクトリは本体構成に対応させる

## 3. プロジェクト全体に通底する開発ポリシー

### 3.1 DRY 原則

- 同一ロジックの重複実装を禁止する
- 共通化は「意味単位」で行い、過剰抽象化は避ける
- 設定値・定数・変換表は1か所に集約する

### 3.2 オブジェクト指向 / 設計原則

- 単一責任（SRP）を守る
- 依存性逆転（DIP）を意識し、具体実装へ直接依存しない
- 拡張時に既存コードの変更範囲が最小となる設計を優先する（OCP寄り）
- 状態を持つコンポーネントはライフサイクルを明示する（生成・再利用・破棄）

### 3.3 レイヤ分離

- Presentation（API/UI）と Application（ユースケース）を分離
- Application と Infrastructure を分離
- 例外変換は境界層で実施し、内部例外を外に漏らさない

### 3.4 設定駆動

- 環境差分はコードではなく設定で切り替える
- 設定は型付きで検証し、不正値は起動時に失敗させる（Fail Fast）
- 秘密情報はリポジトリに置かず、注入元を明確化する

### 3.5 エラーハンドリング

- 予測可能な失敗は明示的にハンドリングする
- ログは「原因特定に必要な最小情報」を構造化して出力する
- APIでは内部詳細を隠し、利用者向けエラーを返す

### 3.6 テスト戦略

- Unit: ビジネスロジックの正しさ
- Integration: 外部依存との接続と契約
- E2E: ユーザーシナリオ成立性
- 変更時は最低でも影響範囲の Unit/Integration を更新する

### 3.7 コード品質ゲート

- Formatter, Linter, Type Checker, Test をCIで必須化
- `main` へのマージ条件を明文化する
  - 例: 全テスト成功、静的解析エラー0、レビュー承認

## 4. その他特記事項（移植時チェックリスト）

- 依存関係の固定方針を決める（lockファイル運用）
- ローカル実行と本番実行の差分を文書化する
- 性能要件（レイテンシ/スループット）と計測方法を明記する
- セキュリティ要件（認証/認可/監査ログ）を初期段階で定義する
- 監視要件（ログ保持期間、アラート条件）を運用前に確定する

運用ルール:

- このテンプレートを新規プロジェクトにコピーしたら、`<>` プレースホルダを必ず埋める
- 固有情報を追加する場合も「なぜ必要か」を一行で追記する
- 方針変更時はドキュメントを先に更新し、実装と同期させる
